TRUST MODEL

安装完成不等于可进入工作台

  • 必须验证选定 Gateway 的运行模式与身份
  • 必须完成认证或管理员配对
  • 默认模型需要真实调用成功

PERSISTENCE

两类持久状态

  • Rust Bootstrap：目录、runtime、迁移事务
  • Renderer localStorage：setup-done 与 UI 偏好

INSTALL LATENCY

最短安全等待

  • Node release index 与 checksum authority 并发解析
  • 现有环境、npm cache 与 SHA-256 验证的 runtime 下载缓存优先复用
  • Windows x86 复用稳定 MinGit fallback，缺失才下载；完整性与事务门禁不省略

RECOVERY

失败时保守回退

  • 目录事务可恢复或回滚
  • 运行模式切换可回滚
  • Gateway 失效重新进入检测
FIG. 02

系统架构检测与 Native / Docker 运行方式确认

安装包先与 Windows 架构匹配；首次启动后 Rust 根据编译目标自动得到 windows/x86、windows/x86_64 或 windows/aarch64，无需用户选择架构。x86 自动进入 Native-only：复用或下载 win-x86 Node.js（npm 随 Node 提供）与 32-bit MinGit，并禁用无意义的 Docker 探测。x64/ARM64 可继续确认 Native 或 Docker。等待时间策略为：并行执行互不依赖的只读检测，优先复用已验证环境与安全缓存，缺失才下载、损坏才修复；最终仍合并到同一个 Gateway 验证门。

WINDOWS x86 · 32-BIT

自动走 Native-only

  • ARCH=x86，不需要用户选择
  • Node release catalog 自动选择 win-x86 兼容版本，npm 随 Node 提供
  • 使用稳定 JunQi-owned 目录复用已固定 SHA-256 的 32-bit MinGit
  • Docker 被能力门禁禁用，并显示“32 位仅支持 Native”而非误报“未安装”

WINDOWS x64

Native 或 Docker

  • ARCH=x86_64
  • Native 使用 win-x64 Node 与 64-bit Git
  • Docker 仍需 CLI、Daemon、镜像与容器契约全部通过

WINDOWS ARM64

架构对应运行时

  • ARCH=aarch64
  • Native 使用 win-arm64 Node 与 ARM64 Git
  • Docker 可用性以真实 ARM64 Docker 环境探测为准

AUDIT

关键风险处置状态

“已修复”表示源代码与自动化回归已经通过；不等于真实环境验收。FIG. 03 本轮变更仍需在 Windows、真实 OpenClaw Gateway、设备配对及 operator.admin scope 升级环境中验证。

已修复 · CONCURRENCY

开机自启交接期间锁定 Ready 导航

自启动组件会将 busy 状态提升到 Ready 页；系统服务交接期间“上一步”和“进入仪表盘”均禁用，避免设置页与工作台同时操作 Gateway 生命周期。

代码已修复 · WIZARD HANDOFF FENCING

过期 Wizard 完成任务不得提交交接结果

official-service handoff、交接后 Gateway probe、模型 live probe 与连接目标刷新之后均重新验证同一 Wizard operation identity。页面返回或新操作接管后，旧任务即使完成 Rust 交接，也不得修改设置 UI、进入 Ready 或继续后续探测。待 Windows 官方服务交接真机实测。

代码已修复 · RUNTIME CREDENTIAL / PROBE FENCING

同端口 Native/Docker 不再共享设备凭据或完成条件

后端为 selected runtime/config 输出非敏感 credential scope；设备 token 的预认证存储键绑定 scope + endpoint，Native 与 Docker 即使共用 127.0.0.1:18789 也不会读取对方 token，且 selected runtime 不再回退到无 scope 的旧 endpoint 凭据。模型 live probe 固定使用启动时的 selected config，并在 CLI 返回后复核 runtime mode 与 config path。待真实配对、模型调用和切换竞态实测。

代码已修复 · ATTESTED CREDENTIAL PROMOTION

selected-scope token 仅向当前已证明 instance 提升

collaboration instance 可用后，迁移事务读取 selected runtime token，先写入 instance key，再发布非敏感 endpoint alias，最后删除旧安全副本；每个不可逆边界均复核 verified connectionId、endpoint 与 runtimeId。身份漂移时不发布 alias、不删除来源，允许保留安全重复副本。待真实 Gateway instance 与 token rotation 实测。

代码已修复 · TOKEN ROTATION

已绑定 instance 的 rotation 写回 instance slot

endpoint alias 已指向 instance 后，Gateway 返回的新 device token 不再回写 selected-scope 或裸 endpoint slot，而是写入当前 durable instance key；避免下一次 promotion 用旧 instance token 覆盖新 token。已通过源码契约测试，待真实 Gateway token rotation 实测。

已修复 · PERMISSION UX

通知权限延后到首次设置完成

通知权限现在只会在 setupComplete 提交后延迟请求，不再打断语言、存储、安装或 Gateway 授权步骤。后续可选优化是改为明确用户手势触发。

代码已修复 · WIZARD CANCELLATION

保留被 OpenClaw 锁定取消的 Wizard 会话

当官方 wizard.cancel 返回 status: running 时保留 sessionId，禁止创建竞争会话并允许继续 resume。自动化测试已通过；尚未在真实 Gateway 持久化写入阶段触发并验证。

代码已修复 · AUTHORIZATION DIAGNOSTICS

保留 OpenClaw 结构化 scope 错误

Gateway 现在保留 details.missingScope 与 details.requiredScopes，并将 MISSING_SCOPE 与 PAIRING_REQUIRED 分开传播；避免把 scope 拒绝误报成设备配对。已通过模拟协议响应测试，待真实 Gateway scope upgrade 实测。

代码已修复 · SECRETREF CONNECTION

连接前通过 OpenClaw 官方 resolver 解析 bootstrap token

selected openclaw.json 未提供 literal token 时，FileReadResolver 调用 get_gateway_token，并由 OpenClaw secrets.resolve 临时解析 gateway.auth.token；解析失败仍保留 selected endpoint，禁止回退到旧 runtime 缓存。自动化测试已通过，待真实 env/file/exec SecretRef 后端验证。
