# Windows/macOS 运行时聚合修复规范

## BUG-RP-01 · 更新操作必须改变活动托管运行时

**当前**：系统工具可用时仍能触发托管更新，下载后的默认托管副本不会成为活动工具。

**目标**：托管更新只对 `managed`、`custom` 或不可用工具开放；活动系统工具明确交给系统更新。

**验收**：

- [x] 可用系统 Node/Git 不暴露托管更新命令。
- [x] 默认托管或自定义运行时可以更新原目录。
- [x] 工具不可用时可以通过托管安装恢复。

## BUG-RP-02 · 单一 Node 平台制品模型

**当前**：索引制品键和下载归档名分别拼装。

**目标**：一个平台模型同时描述支持能力、索引键、归档格式、文件名和顶层目录。

**验收**：

- [x] Windows x64/ARM64 映射 ZIP 制品。
- [x] macOS Intel/ARM64 映射 tar.gz 制品。
- [x] 不支持的平台或架构明确失败。

## BUG-RP-03 · 单一下载源目录

**当前**：真实下载源和 UI 顺序重复维护。

**目标**：Node/Git 的 URL、日志名称和 UI 显示名称均从各自唯一源目录生成。

**验收**：

- [x] 索引、SHA-256、归档和状态展示顺序一致。
- [x] Node 不包含 nodejs.org，Git 不包含 GitHub。
- [x] 增删源只需修改一个目录定义。

## BUG-RP-04 · 明确运行时来源

**当前**：`local` 同时表示自定义目录和默认托管目录。

**目标**：后端返回 `system`、`managed`、`custom`，前端逐项显示。

**验收**：

- [x] 自定义目录显示“用户选择的目录”。
- [x] 默认托管目录显示“JunQi 托管”。
- [x] 不再产生或消费 `local` 来源。

## BUG-RP-05 · 单一平台能力模型

**当前**：存储、状态和安装模块分别判断平台能力。

**目标**：产品能力由集中模型提供，条件编译仅保留实现层边界。

**验收**：

- [x] Windows 支持托管 Node 和 Git。
- [x] macOS 仅支持托管 Node。
- [x] Linux 两者均由系统管理。
- [x] 存储选择和设置更新使用同一能力结果。
